# PRODUCT.md — Operon

Product strategy: what Operon is, who it serves, how the sale and EPP work, and where the roadmap is headed.

**When to consult:** When writing UI copy, making tone decisions, deciding what requires KYC, or building anything that touches the sale mechanics, referral rules, or EPP programme.

**When to update:** When product rules change (commission rates, tier thresholds, invite policy, programme structure) or when the roadmap shifts phases. Do not use this file for implementation detail — that belongs in `ARCHITECTURE.md`.

---

## What Operon Is

**Operon** is a DePIN-adjacent node-sale platform. The product is a **Genesis Node Sale**: 100,000 network participation licences represented as ERC-721 NFTs, minted across Arbitrum and BNB Smart Chain, sold through a 40-tier price curve, with a multi-level referral programme built into the purchase flow.

The dashboard (`app.operon.network`) is a single application serving three overlapping user roles — buyers, community referrers, and Elite Partners — from one account and one wallet connection. There is no separate sale page and dashboard.

The $OPRN token is **not** live at sale time. Nodes are sold as participation licences. Emissions and token rewards begin at a later phase (see Phase Timeline).

---

## Target Users

Three overlapping roles. A single user can be all three.

| Role | How acquired | What unlocks |
|---|---|---|
| **Visitor** | Lands on the dashboard, may or may not connect a wallet | Sale page (view only), public info |
| **Node buyer** | Purchases ≥1 node | Node inventory, rewards tab (post-emissions), auto-generated personal referral code (`OPR-XXXXXX`) |
| **Community referrer** | Any wallet on first connect (automatic) | Referrals tab: share their personal code, buyers get 10% off, they earn level-based commission |
| **Elite Partner (EPP)** | Invited via a private `EPP-XXXX` link, onboarded at `/epp/onboard` | EPP referral code (`OPRN-XXXX`), buyers get 15% off, full commission waterfall + milestone bonuses + tiered stipends, post-TGE reward pool participation |

**Every connected wallet gets a personal community code** generated at first signin. Buying a node is not required to earn commission — referral activity and purchase activity are decoupled.

## Sale Mechanics

### Supply

- **100,000 total nodes** across 40 price tiers
- Supply is **split between Arbitrum and BNB Smart Chain** but the quota is **collective** — Tier 1 has 1,250 nodes available across both chains combined, not 1,250 per chain
- **Overflow is accepted** — if Arbitrum and BSC both sell into the same tier simultaneously, a small overflow is allowed rather than trying to hard-coordinate across chains (low expected velocity)
- Contract has **split supply**: `publicSupply` (purchaseable) + `adminSupply` (admin-mintable, never overlaps with public)

### Pricing

- 40 tiers, each tier priced higher than the last
- Current seed: Tier 1 at $500, +5% per tier
- Stablecoin-only payment: USDC or USDT, on either chain
- Token decimals: 6 on Arbitrum, 18 on BSC. All money stored as **USD cents (integer)**, never float — see `ARCHITECTURE.md` and `ALGORITHMS.md`

### Discounts

Three discount levels, fixed at launch:

- **0%** — no referral code
- **10%** — community referral code (any wallet's personal `OPR-XXXXXX`)
- **15%** — EPP referral code (`OPRN-XXXX` from an Elite Partner)

Discounts apply at any tier, at any time during the sale. Applied on the final purchase price and recorded in `purchases.discount_bps`.

### Sale Stages

- `active` · `paused` · `closed`
- **No "whitelist" stage** — this was removed in migration `008_product_changes.sql`
- Stage lives in the single-row `sale_config` table and is controlled by `/api/admin/sale/pause` / `/unpause`, which call the contracts via the admin signer (see `OPERATIONS.md`)
- Sale stage propagates to all connected clients via Supabase Realtime

### Purchase Flow

1. Buyer lands on the sale page (possibly via `?ref=CODE`, captured into sessionStorage)
2. Selects chain (Arbitrum or BSC), quantity, payment token (USDC/USDT)
3. Enters referral code (or URL-captured code is prefilled)
4. Approves the exact token amount (never unlimited approval)
5. Calls `purchase()` on the NodeSale contract
6. Frontend waits for ≥1 block confirmation before showing "purchase successful"
7. Webhook (Alchemy for Arbitrum, QuickNode for BSC) delivers the `NodePurchased` event to our API
8. We re-verify the event on-chain, then process referral attribution + tier counter atomically via the `process_purchase_and_commissions` Postgres RPC
9. Reconciliation cron every 5 minutes catches any missed webhooks

## Elite Partner Programme

### Invitation Only

EPP is gated. There is no public signup. Invites are pre-generated by admin (`POST /api/admin/epp/invites`) and distributed manually as URLs:

```
https://operon.network/epp/onboard?inv=EPP-XXXX
```

The optional `&name=David` parameter personalises the letter on step 0. Without it, the letter defaults to "Dear Partner,".

Invites currently have **no expiry**. Decision pending in `DECISIONS.md` on whether to set a 14- or 30-day default.

### Onboarding Flow

4 steps at `/epp/onboard`:

1. **Letter** — the personal invitation (styled as an exclusive serif/gold aesthetic, distinct from the dashboard)
2. **Terms & Conditions** — 9 accordion sections, 3 required confirmation checkboxes (accept T&Cs, not a US person, nodes are participation licences not securities)
3. **Wallet + Form** — email (required), payout chain selector, Telegram (optional), RainbowKit wallet connect. Signs a SIWE message to prove wallet ownership.
4. **Confirmation** — generated `OPRN-XXXX` referral code, copy / share / open dashboard

The connected wallet becomes both the authentication wallet and the payout wallet. Changing the payout wallet later requires contacting support.

All 6 languages are translated (EN, TC, SC, KO, VI, TH). Thai is real prose, not a stub — legal review pending.

### Commission Waterfall

**9 levels deep**, rates determined by each upline's **own EPP tier** at the time of the purchase:

| Tier | L1 | L2 | L3 | L4 | L5 | L6 | L7 | L8 | L9 |
|---|---|---|---|---|---|---|---|---|---|
| Affiliate | 12% | 7% | 4.5% | 3% | — | — | — | — | — |
| Partner | 12% | 7% | 4.5% | 3% | 2% | — | — | — | — |
| Senior | 12% | 7% | 4.5% | 3% | 2% | 1.5% | — | — | — |
| Regional | 12% | 7% | 4.5% | 3% | 2% | 1.5% | 1% | — | — |
| Market | 12% | 7% | 4.5% | 3% | 2% | 1.5% | 1% | 0.75% | — |
| Founding | 12% | 7% | 4.5% | 3% | 2% | 1.5% | 1% | 0.75% | 0.5% |

**Referral and purchase activity are decoupled** — an EPP partner who has never bought a node still earns full commission on their downline's purchases. Level 10+ uplines get nothing (hard cap).

**Self-referral is blocked** by same-wallet equality check at signup time. Detected post-facto self-referral invalidates referral rewards. See `ALGORITHMS.md` for chain-walk semantics.

### Credited Amount & Tier Advancement

Each level contributes a weighted fraction of the purchase to the upline's `credited_amount`:

L1: 100% · L2: 25% · L3: 10% · L4: 5% · L5: 2.5% · L6–9: 1%

When `credited_amount` crosses a threshold, the partner is **instantly promoted**:

| Tier | Threshold |
|---|---|
| Affiliate | $0 |
| Partner | $5,000 |
| Senior | $25,000 |
| Regional | $100,000 |
| Market | $250,000 |
| Founding | $1,000,000 |

Promotion is one-way from auto-advancement. Manual demotion (or any manual override) requires `/api/admin/partners/tier` with a written reason. See `OPERATIONS.md`.

### Milestone Bonuses

One-time USD bonuses paid at accumulated credited-amount thresholds:

| Threshold | Bonus |
|---|---|
| $10,000 | $500 |
| $25,000 | $1,500 |
| $50,000 | $5,000 |
| $100,000 | $15,000 |
| $250,000 | $50,000 |
| $500,000 | $90,000 |
| $1,000,000 | $150,000 |

Paid at the biweekly settlement cycle, not instantly.

### Payout

- Commissions paid **biweekly** in USDC
- Backend **records** payouts but **never sends** them — operator manually sends from a separate wallet, then records the tx hash via `POST /api/admin/payouts/mark-paid`
- Rationale: backend never holds funds, never signs payout transactions. See `DECISIONS.md` D05.

### Stipend

Partners at qualifying tiers receive a monthly stipend paid biweekly in USDC, subject to a minimum sales-activity gate. Full schedule provided to partners upon acceptance (not in public docs).

## Phase Timeline

### Phase 1 — Genesis Sale (current, mostly shipped)

- Node sale across Arbitrum + BSC
- EPP invite flow + onboarding
- Community referral programme with auto-generated personal codes
- Commission waterfall + tier auto-advancement + milestone bonuses
- Admin panel for ops (pause/unpause, event replay, tier overrides, payout marking, invite generation)
- Multi-language dashboard (6 languages)

**Revenue:** Genesis sale proceeds.
**What's live:** Testnet. Mainnet contract deploy pending.

### Phase 2 — Emissions + Staking (next)

- Network emissions start — node holders begin accumulating $OPRN-denominated rewards (held in contract, claimable at TGE)
- Node uptime tracking + reward multipliers
- Staking of node NFTs for boosted rewards
- Node delegation (operator-run node-as-a-service, third-party partnerships)
- Dashboard surfaces: rewards tab, staking UI, delegation management

**Open decisions for Phase 2** — see `DECISIONS.md` D-pending section. All Phase 2 API routes, contracts, and schema additions are pre-stubbed in `ARCHITECTURE.md` → "Phase 2 Surface".

### Phase 3 — TGE + Post-TGE

- Token Generation Event
- Claim portal — holders convert accumulated rewards into on-chain $OPRN
- **Post-TGE commissions paid in $OPRN from the Referral and Distribution Pool**, not USDC
- Secondary market opens (assuming transfer lock expires — currently set at 12 months)
- Node Ops / delegation marketplace (third-party integrations)

## Critical Product Rules

These show up in UI copy and must never be violated:

1. **Nodes are participation licences, not securities or investments.** Never in UI copy, partner comms, or marketing materials.
2. **No guaranteed returns.** Rewards are variable and depend on network performance.
3. **No US persons targeted for the token component.** Reinforced in EPP T&C checkbox flow.
4. **Community codes are generated for every connected wallet** (CLAUDE.md rule 8) — not only after purchase.
5. **Never show "purchase successful"** until ≥1 block confirmation.
6. **All user-facing strings go through `t()`** — no hardcoded text, 6 languages must stay in sync.
7. **EPP T&Cs may change with prior notice.** Continued participation after an amendment constitutes acceptance — this is section 9 of the T&Cs in `app/epp/onboard/epp-translations.ts`.

## User Experience Principles

- **Medical language avoided** — Operon is a participation licence, not a health product. (This rule is carried from the Trace project's convention about tone, adapted: never use clinical/regulated framing.)
- **Partners are sophisticated.** EPP onboarding is styled as a letter because partners are typically recruited through personal relationships. The tone is deliberate: "We would like to invite you" not "Sign up now".
- **Buyers come from many backgrounds.** Sale UI is in plain language, minimal jargon, with the 6-language selector always visible.
- **Dashboard density, not spreadsheet density.** Sale status + referral code + purchase flow should fit on one screen. The old "separate sale page, dashboard, and claim portal" split (seen in competitor projects like Aethir) is intentionally avoided — one app.
